# Handbook: No AI/IDE Attribution Banners in Product Artifacts

**Scope:** all PRs (handbook lives under `general/` — Rex always loads it).
**Enforcement:** advisory.

## The rule

Tool-attribution banners — "Made with Cursor", "Generated by Claude", "Built with Copilot", "Powered by v0", "Created with Bolt", and similar — do **not** belong in **product artifacts**. Product artifacts are anything a user or reader sees as the finished thing: README headers/footers, landing-page and site copy, generated HTML/CSS output, UI footers and about screens, `<meta name="generator">` tags, email templates, doc-site chrome, and CLI banner/version output.

The tool used to author the code is not a product feature and does not need marketing credit inside the deliverable. Strip the banner; keep the work.

## Why

An attribution banner in a shipped artifact is noise that quietly degrades the product:

- **It reads as unfinished.** A "Generated by X" footer signals boilerplate scaffolding nobody cleaned up — the opposite of the polish the artifact is supposed to convey.
- **It's unwanted advertising.** The banner markets a third-party tool to the project's users, on the project's surface, without the project's consent.
- **It leaks the toolchain.** Which IDE/model produced a file is internal context, not something to broadcast on a public README or site.

None of this is catastrophic — a stray banner has no material downstream cost (no data loss, no outage, no security incident), which is exactly why this handbook is **advisory**, not blocking. Rex surfaces it as a `nit:` / `suggestion:` so it gets cleaned up without churning the merge gate.

## What Rex flags

Surface an advisory finding when a PR **adds** any of the following to a product-facing artifact:

1. Attribution phrases in visible copy — `Made with <tool>`, `Generated by <tool>`, `Built with <tool>`, `Created with <tool>`, `Powered by <tool>`, `Written by <AI model>` — where `<tool>` is an AI/IDE codegen tool (Cursor, Copilot, Claude, ChatGPT/GPT, Codeium/Windsurf, v0, Bolt, Lovable, Devin, Tabnine, etc.).
2. A `<meta name="generator" content="...">` tag naming an AI/IDE tool in generated HTML.
3. A README badge or footer line crediting the codegen tool (e.g. a shields.io "made-with-cursor" badge, or a `> Generated with Claude` blockquote in a project README).
4. CLI/app banner or `--version` output that appends "built with <tool>".
5. HTML comments left in shipped markup that credit the generator (`<!-- Generated by v0 -->`).

Be specific about the surface in the finding — name the file and the line so the author can strip just the banner.

## Sample finding

> **No AI-attribution banners** — `README.md` line 3 adds a `> 🤖 Generated by Cursor` blockquote under the project title. The authoring tool isn't a product feature; remove the banner so the README reads as the finished project. (advisory — `suggestion:`)
>
> **No AI-attribution banners** — `public/index.html` adds `<meta name="generator" content="Built with Copilot">`. Drop the generator meta tag; it advertises the toolchain on a user-facing page. (advisory — `nit:`)

## What's NOT a violation

- **The framework's own required trailers.** ApexYard **requires** the `Co-Authored-By: Claude ...` trailer on commit messages and the `🤖 Generated with [Claude Code]` trailer on PR bodies (see `CLAUDE.md` § Git Conventions and `.claude/rules/git-conventions.md`). These are process metadata on commits and PRs — **not** product artifacts — and are mandated by the framework. **Never flag them.** A commit trailer, a PR-description trailer, or their presence in `git log` output is expected and correct.
- **Genuine authorship credit for humans or the project itself** — `Made by <team>`, `© <company>`, a real `AUTHORS` file, a human `Co-Authored-By:` pair line.
- **Documentation that discusses these tools as subject matter** — a blog post, changelog, or handbook (like this one) that mentions "Cursor" or "Claude" while writing *about* them, rather than crediting them as the generator of the surrounding artifact.
- **Dependency/attribution notices required by a license** — a `NOTICE` file or third-party-license section is a legal obligation, not a vanity banner.
- **Internal-only scaffolding comments** that never ship to a user-facing surface and are clearly transient (though cleaning them up is still good hygiene).

When in doubt, ask: *is this on a surface the product's user or reader sees as the finished thing?* If no (a commit, a PR body, a license notice), it's not a violation.
