## [SYSTEM_FLOW]

The operational source of truth is the Visit. Every physical person entering the workspace gets a Visit record. Billable space use gets a Usage Session. Sales-like activity is not a standalone POS flow; it is captured as Charges/Add-ons attached to an existing operational target.

```text
Physical arrival
  -> Visit
  -> optional Usage Session when occupying a space
  -> Charges/Add-ons attached to visit/session/event/host/invoice draft
  -> Checkout calculates usage, coverage, billing responsibility, and charges
  -> Invoice(s) generated by responsibility
  -> internal Payment(s) recorded when required
  -> Visit/session close
  -> Space becomes cleaning, maintenance, or available
```

Visitor types:

- `walk_in`
- `member`
- `booking_customer`
- `hosted_guest`
- `event_attendee`
- `non_billable`

Billing responsibility values:

- `visitor_pays`
- `host_pays`
- `company_pays`
- `event_pays`
- `subscription_included`
- `complimentary`
- `pay_later`

Payment methods are internal records only:

- `cash`
- `card_terminal`
- `wallet`
- `bank_transfer`
- `instapay`
- `mixed`
- `pay_later`
- `host_account`

## [INTERACTION_POLICY]

Backspace's rich UI direction is operational, not decorative. A screen is "rich" when it makes staff work faster, safer, and clearer: visible state, keyboard-friendly actions, precise feedback, and fewer repeated decisions.

Allowed tools, only when the specific implementation issue needs them:

- `motion` for purposeful route, drawer, state-change, and success/error feedback. No marketing choreography, bounce, parallax, or decorative page animation.
- `cmdk` for real command/search workflows: finding people, visits, bookings, add-charge targets, checkout, and quick actions.
- `@tanstack/react-table` for dense operational tables that need sorting, filtering, column visibility, row selection, pagination, or bulk workflows. Do not hand-roll complex tables.
- `recharts` for reports only when a chart answers a named management question: occupancy, revenue, receivables, usage, or trend comparison. No vanity charts.
- `@dnd-kit/*` only for real reorder/assignment flows such as cleaning queue priority or task assignment. No decorative drag/drop.
- `dotLottie` only for useful empty/success states where it improves clarity. It is not a default success-screen pattern.

Frontend implementation rules:

- Use TanStack Router for route structure and protected staff routes.
- Use TanStack Query/tRPC for server state, cache invalidation, stale-state handling, and mutation feedback.
- Use TanStack Form with Zod-backed validation for complex operational forms and drawers.
- Use shadcn/ui primitives first; add shared components in `packages/ui` only when reuse is real.
- Design forms, tables, drawers, status badges, command/search, and reports around staff operations density, not generic SaaS templates.
- Any PR that adds a persistent UI dependency must explain why in the PR body and include an AgDR when the dependency becomes a product/architecture decision.

## [DOMAIN_MODEL]

Core persistence areas:

- Workspace topology: branches, floors, spaces, space status history.
- People and accounts: people, customer accounts, tenants, hosted guest links.
- Subscriptions: plans and memberships.
- Scheduling: bookings, events, event attendees.
- Daily operations: visits and usage sessions.
- Add-ons and billing: catalog items, charges, invoices, invoice items, payments, refunds, shifts.
- Facilities: cleaning tasks and maintenance tickets.
- Governance: staff profiles, roles, permissions, branch access, approvals, audit logs.

Critical invariants:

- Money is stored and calculated in integer minor units.
- Paid/finalized invoices are immutable.
- Corrections use adjustment, refund, reversal, or credit-note style records.
- Cash payments require an open shift.
- No double booking or check-in to occupied, maintenance, blocked, or inactive spaces.
- Sensitive operations write audit logs server-side.
- UI authorization is convenience only; server permission checks are authoritative.
