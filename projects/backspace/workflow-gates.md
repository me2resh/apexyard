**Project**: `backspace`

This checklist records how the Backspace planning package maps to the ApexYard SDLC now that the parent epic and initial implementation issues are filed. It also records the interaction/dependency policy that future implementation PRs should follow.

## Scope Boundary

- Existing app only: `workspace/backspace` generated by Better-T-Stack.
- Fixed stack: pnpm, React, TanStack Router, Hono on Node, tRPC, Better Auth, PostgreSQL, Drizzle, Docker DB, Tailwind CSS, shadcn/ui, evlog, lefthook, skills, Vite+.
- No new scaffold, no base-stack replacement, no payment provider, no deployment setup.
- Product rule: no standalone anonymous POS as the normal flow. Sales-like activity is modelled as contextual Charges/Add-ons attached to Visit, Usage Session, Event, Hosted Guest, Tenant/Host Account, or Invoice Draft.
- Rich interaction rule: UI richness means operational clarity, speed, feedback, and state visibility. Optional tools are allowed only when a ticket needs them: `motion`, `cmdk`, `@tanstack/react-table`, `recharts`, `@dnd-kit/*`, `dotLottie`.

## Interaction / Tooling Gate

- Default UI stack stays React, TanStack Router, TanStack Query, TanStack Form, Tailwind, and shadcn/ui.
- Forms should use TanStack Form plus Zod-backed validation for complex operational flows.
- Complex tables should use `@tanstack/react-table` when table behavior is central.
- `cmdk`, `motion`, `recharts`, `@dnd-kit/*`, and `dotLottie` are permitted only when the active ticket's workflow needs them.
- PRs that add persistent UI dependencies must explain the operational need and add an AgDR when the dependency is a broad product/architecture decision.
