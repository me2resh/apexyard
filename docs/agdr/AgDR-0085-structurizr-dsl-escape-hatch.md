# Structurizr DSL as a dependency-free escape hatch for L3+ C4 diagrams

> In the context of `/c4`'s Mermaid-only C4 output hitting a real ceiling at L3 (component) precision — the exact gap AgDR-0003 explicitly deferred — facing the choice between shelling out to a Structurizr CLI/Docker renderer, bundling a renderer, or emitting Structurizr DSL as plain text, I decided to **emit Structurizr DSL text only** (`workspace.dsl`, generated by a new `/c4 --dsl` mode) to achieve L3+ capability without adding any new runtime dependency to the framework, accepting that rendering the `.dsl` into a viewable diagram is left entirely to the adopter's own tooling choice.

## Context

AgDR-0003 shipped Mermaid C4 as the v1 diagramming format for `/c4` and explicitly deferred Structurizr DSL: *"Structurizr DSL as an escape hatch for projects that hit Mermaid's C4 ceiling (L3 component diagrams, precise 'technology' metadata, auto-zoom). Add when a project asks."* Issue #68 filed that follow-up with an explicit "NOT NOW — open only when a managed project explicitly asks" gate. This AgDR is that decision, made when scoping the ticket's implementation.

The gap is real and unchanged from AgDR-0003's analysis: Mermaid's C4 support is "beta," has no auto-zoom between levels (L1/L2/L3 are hand-maintained, independently-drifting files), and gets visually unwieldy once component-level detail enters the picture. Structurizr is the industry-standard C4 modelling tool and solves exactly this — one `model { }` block, multiple auto-generated `views { }`.

The open question this AgDR resolves is narrower than "should we support L3?" (issue #68 already answers yes) — it's **how the framework should ship that support** without repeating the cost profile AgDR-0003 rejected Structurizr for in the first place: a Java/Docker runtime dependency and GitHub not rendering the format inline.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Shell out to `structurizr-cli` (Java)** at generation time — the skill renders PNG/SVG directly | Adopter gets a finished image, zero extra steps | Requires bundling or requiring a Java runtime the framework has never depended on; breaks on any machine without Java; violates the framework's zero-install posture that made Mermaid win in AgDR-0003 in the first place |
| **Bundle a renderer** (vendor a WASM Structurizr port, or a Docker image the skill invokes) | Self-contained; no adopter setup | New maintenance surface (keeping a vendored renderer current); Docker-in-the-loop is exactly the two-source PNG+DSL problem AgDR-0003 rejected; large scope increase for a "when a project asks" feature |
| **Emit Structurizr DSL as text only** — `workspace.dsl`, no render step performed by the framework | Zero new runtime dependency (pure Bash/awk for generation + structural lint); the adopter renders with whatever they already have or prefer (structurizr.com/dsl online editor needs literally nothing installed); matches the framework's existing template-file pattern (`.md` in, filled `.md` out) — here it's `.dsl` in, filled `.dsl` out | The adopter doesn't get a rendered image for free; GitHub doesn't preview `.dsl` inline (same trade AgDR-0003 already flagged and accepted as a cost of the *escape hatch*, not the default) |

## Decision

Chosen: **emit Structurizr DSL as text only**, because it is the only option that doesn't reintroduce the exact runtime-dependency cost AgDR-0003 weighed Structurizr against and rejected for the *default* path. The escape hatch inherits that same constraint — it exists for the rare project that needs L3+, not to change the framework's zero-install posture.

Concretely:

- New `/c4 <project> --dsl` mode. Reuses the skill's existing detection (steps 2–3: containers, external actors, relationships) unchanged; only the output-generation step branches to DSL instead of Mermaid.
- New template `templates/architecture/c4-structurizr.dsl` — the DSL-equivalent of `c4-context.md` / `c4-container.md`, using the same worked fictional example (User/Admin, Auth/Payment/Email providers, Web/API/Worker/DB/Cache containers) so the three templates read as one family, plus an L3 `component { }` block inside the `api` container as a worked example of the detail Mermaid can't express.
- New structural lint `.claude/skills/_lib-structurizr-lint.sh` (+ `c4/lint-dsl.sh` wrapper) — validates the emitted DSL with **zero external tools**: balanced braces, presence of `workspace` / `model { }` / `views { }`, no duplicate identifier assignments. If the adopter happens to have `structurizr-cli` on PATH, the lint additionally attempts a real parse via it — a bonus, never a requirement, and a Java-runtime-missing failure from that optional path degrades back to the structural-only result rather than blocking.
- Mermaid stays the unconditional default for L1/L2 — nothing about the existing `/c4` invocation shape changes. `--dsl` is additive.
- L3 (component) detection is explicitly **not** automated in this change — it would require call-graph-level source analysis, a materially larger feature than this ticket scoped. The generated `workspace.dsl` auto-fills L1+L2 from the same detection as Mermaid and gives the adopter a worked L3 example to extend by hand.
- Rendering is entirely the adopter's choice, documented but not automated: the free [structurizr.com/dsl](https://structurizr.com/dsl) online editor (zero install), Structurizr Lite (Docker), or `structurizr-cli` (Java) if they already have one.

## Consequences

- `/c4` gains a real answer to "we hit Mermaid's L3 ceiling" without the framework taking on a Java or Docker dependency anywhere in its own generation or lint path.
- The three architecture templates (`c4-context.md`, `c4-container.md`, `c4-structurizr.dsl`) now share one worked example across Mermaid and DSL, making it easy to see they describe the same model in two formats.
- Deferred, not filed as separate follow-up tickets (matching AgDR-0003's own "add when a project asks" pattern): auto-detected L3 component inference from source; committing rendered PNG/SVG alongside the `.dsl` (issue #68's raw checklist mentioned this — skipped here because it reintroduces the two-source problem for an escape hatch whose whole point is staying text-only; an adopter who wants a committed image can render and commit it themselves); a `/decide`-style automated prompt comparing Structurizr vs "just draw another Mermaid" (the "When to reach for `--dsl`" table in the `/c4` SKILL.md serves this purpose today without new automation).
- Low cost if wrong: the DSL path is fully additive. If no project ever uses `--dsl`, the cost is one template file, ~180 lines of lint script, and a SKILL.md section — deleting all three is one commit, same "no lock-in" profile AgDR-0003 claimed for Mermaid.
