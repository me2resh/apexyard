# opencode Gate Adapter — Settings.json-Derived Dispatcher Over the Bash Hooks

> In the context of making apexyard's governance gates enforceable inside opencode (opencode.ai) as well as Claude Code and pi, facing the choice between reproducing the pi adapter's hand-maintained `DEFAULT_GATES` table or deriving the gate list directly from `.claude/settings.json`, I decided to build a single dispatcher plugin that parses `.claude/settings.json` at plugin-load time and shells out to the existing, unmodified bash hooks to achieve zero logic duplication AND zero gate-table drift, accepting the ongoing cost of every derived hook running as a real subprocess per matching tool call (more hooks run per call than pi's curated table would run) and the residual, currently-unverified risk that this build's shipped, settings.json-derived dispatcher was not itself re-verified live against a real, credentialed opencode session (spike #816 proved the pattern live against a throwaway prototype, not this code).

## Context

Spike me2resh/apexyard#816 (promoted from #816, itself following #804's pi precedent) proved — **more decisively than pi** — that an opencode plugin can enforce `block-unreviewed-merge.sh` by registering on opencode's `tool.execute.before` event, reconstructing the exact stdin JSON the bash hook already parses, and mapping the hook's exit code (2 = block, via throw; 0 = allow) to opencode's throw-to-deny contract. The spike ran **fully live**: opencode ships a free, no-API-key model (`opencode/big-pickle`), so a real, model-driven agent turn attempting `gh pr merge` was observed being blocked by the unmodified hook, with a negative control (corrupting the plugin) confirming the block came from the plugin, not opencode itself.

The spike's own scope notes named what should NOT carry over into production: the throwaway prototype plugin and its test harness, any hand-rolled types (opencode's real plugin API should be used, verified against its actual `.d.ts`), and the single-gate proof-of-concept (production needs the dispatcher shape covering the full hook list, not one hardcoded gate).

Feature ticket #821 additionally named a decision this AgDR had to make: whether to reproduce the pi adapter's (AgDR-0082) hand-maintained `DEFAULT_GATES: GateDefinition[]` table, or do something better. That instruction traces to a design-review finding on #730 (the Codex adapter): Tariq flagged that a hand-maintained gate table is a table that can silently drift from `.claude/settings.json`'s real wiring — it only grows when someone remembers to edit it by hand. The Codex adapter (`bin/sync-codex-adapter.sh`, AgDR-0088) had already solved this for its own target format by generating `.codex/hooks.json` directly from `.claude/settings.json` with `jq`, rather than hand-copying hook entries.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Reproduce pi's hand-maintained `DEFAULT_GATES` table** | Fastest to build; identical shape to a proven, shipped adapter; easy to review against a known-good precedent | Carries forward the exact drift risk #730's review flagged — the table only covers gates someone remembered to add, and pi's own table already defers 7 blocking hooks (`validate-branch-name.sh` and others) as "known not-yet-bridged" purely because nobody wired them yet, not because they're structurally hard to wire |
| **Statically generate a gate-table file, mirroring the Codex adapter's `bin/sync-codex-adapter.sh`** | Same convergence property as Codex; a generated file can be diffed/reviewed like source | opencode plugins are plain TypeScript that already runs at session-start time and can read JSON directly — a static generation step (and the accompanying `--check` drift-detection machinery Codex needs because its target format, TOML/JSON consumed by a non-Node runtime, can't read `.claude/settings.json` live) is unnecessary ceremony for a target that CAN read the canonical config live |
| **Derive the gate table from `.claude/settings.json` at plugin-load time (CHOSEN)** | Zero drift by construction — a new hook wired in `.claude/settings.json` is picked up with no adapter change at all; simpler than Codex's static-generation approach because opencode doesn't need it; the derivation logic (`derive-gates.ts`) is independently unit-testable against both a hand-written fixture and this repo's own real `.claude/settings.json` | Cannot pre-curate which hooks are "real gates" vs. advisory-only the way pi's table implicitly does — every `PreToolUse` hook for the covered matchers is derived and executed on a matching call, including advisory hooks that never exit 2 (harmless — they just cost a subprocess spawn); more hooks run per tool call than pi's curated subset would run |

## Decision

Chosen: **derive the full gate table from `.claude/settings.json` at plugin-load time** (`harness-adapters/opencode/src/derive-gates.ts` + `src/gate-dispatcher.ts`), because it closes the exact drift risk #730's review named, is simpler to implement than Codex's static-generation approach for a target (opencode) that can already read JSON live, and — per #821's explicit "do better here and note it as the pattern pi should converge to" instruction — produces a MORE complete gate table than pi's hand-maintained one without any additional maintenance burden going forward.

Three supporting decisions, made while building the dispatcher:

- **Ops-root resolution reuses the pi adapter's approach**, ported with one structural change: opencode resolves the ops root ONCE at plugin init (from `PluginInput.directory`/`.worktree`), not per tool call. pi's dispatcher re-resolves per `tool_call` event because pi hands it a fresh `ctx.cwd` on every event; opencode's plugin function is called once per session and never offers a fresher cwd afterward. The `APEXYARD_OPS_ROOT` env-var override name is reused unchanged rather than inventing a second variable — same deliberate divergence from Claude Code's session-pin mechanism the pi adapter already documented (AgDR-0082), and the same consequence: no protection against resolving to an unrelated ops-fork-shaped directory tree (apexyard#381), mitigated the same way (`APEXYARD_OPS_ROOT` set explicitly when running outside the ops fork's own working tree).
- **Real opencode types, verified against the installed package and opencode's own source, not documentation alone.** `Plugin`, `PluginInput`, `Hooks["tool.execute.before"]` are imported (type-only) from `@opencode-ai/plugin`'s real, installed `.d.ts`. Two facts needed verification beyond opencode's own docs (which paraphrase inconsistently on some of these points): the bash tool's registered id is literally `"bash"` (confirmed against `packages/opencode/src/tool/shell/id.ts` in the `sst/opencode` repo — the tool is implemented in a file named `shell.ts` but kept the `"bash"` id for plugin backwards compatibility, per that file's own comment), and the edit/write tools' target-path field is `filePath` (confirmed against `edit.ts`/`write.ts`'s own Effect Schema `Parameters` struct), not `path` the way pi's tool field is named.
- **Blocking is a throw, not a return value.** opencode's `Hooks["tool.execute.before"]` has no `{block: true}` return contract the way pi's `ToolCallEventResult` does — the documented and only way to deny a tool call is to throw inside the handler. `runGateHook`'s fail-closed semantics (exit 2 → block; no numeric exit status at all → block, naming the hook and error; matches Rex's PR #817 finding on the pi adapter) are unchanged in substance, just surfaced as a throw instead of a return value at the dispatcher's outer boundary.

## Consequences

- A new gate wired in `.claude/settings.json` reaches opencode automatically — no `harness-adapters/opencode/` change required, closing the exact drift risk #730's review named for the pi adapter's hand-maintained table.
- The opencode adapter enforces a STRICTLY LARGER gate set than the pi adapter today (pi's `DEFAULT_GATES` explicitly defers 7 blocking hooks as "known not-yet-bridged"; this adapter derives all of them). This is a genuine behavioral difference between the two harness adapters until pi converges to the same derive-from-settings.json pattern — noted here as a recommended follow-up, deliberately NOT done as part of this change (scope discipline: #821 is the opencode adapter, not a pi refactor).
- More hook subprocesses run per matching tool call under opencode than under pi, because the derived table is not curated down to "obviously a gate." Each is a fast bash script; this build did not benchmark the aggregate latency cost.
- The live, model-driven proof this AgDR's context describes (spike #816) was performed against a throwaway prototype plugin, not this shipped, settings.json-derived dispatcher. This build's environment had no opencode model credentials to re-run that exact live check against the final code — `test/smoke-block-unreviewed-merge.sh` closes as much of that gap as is closable without live opencode credentials (real hook, real subprocess exec, real settings.json parse, fixture ops root, negative control), leaving only "does opencode's own internal event dispatch call this handler the documented way during a live model turn" as the one hop still resting on spike #816's earlier proof rather than a fresh one against this code. See `docs/opencode-adapter.md`'s "honest breakdown" table for exactly what is and isn't re-verified, and the closing step once credentials are available.
- The upstream `batch`-tool bypass (opencode's `batch` tool skips `Plugin.trigger()` entirely — anomalyco/opencode#5894) is documented, not fixed. It is an opencode-side gap outside this framework's control; a batched tool call under opencode is ungated regardless of which gate-table strategy this adapter uses.
