# AgDR-0089 — `/eval-agents` scores against frozen ground-truth defect sets, never a prose rubric

> In the context of building `/eval-agents` (#824) to measure the framework's review agents (Rex/Hakim/Tariq), facing spike #825's finding that a text-only LLM-judge rubric is at-chance on the one question that matters — was an approval justified — I decided the judge establishes ground truth once, offline, from real re-review disagreements and confirmed fixes, and scores each fresh agent run by set-overlap against that frozen answer key with **approve-precision** as the headline metric, to get a measurement that actually detects a fluent-but-wrong review, accepting that corpus curation is manual, human-adjudicated work and the starter corpus is too small to be statistically significant.

## Context

Issue #824 proposed an eval harness for Rex/Hakim/Tariq/Naqid: a labeled PR corpus, 0-4 rubric dimensions, an "LLM-judge calibrated against recorded human verdicts." Spike #825 (`docs/spike-825/`, merged in #827) tested exactly that naive shape — a judge scoring review *prose* against a rubric — against 19 real review units mined from this repo's own PR history. The result: aggregate rank-concordance looked fine (Goodman-Kruskal gamma = 0.83), but that number was a mirage. Restricted to the load-bearing case — separating a *correct* approval from a *wrong* one — the rubric was at chance (`GOOD_CLEAN` mean 14.00/16 vs `MISS` mean 13.67/16, and the single most fluent wrong-approval outscored 7 of 10 correct approvals). A surface judge cannot tell a thorough-and-right review from a thorough-looking-but-wrong one, which is precisely the failure mode this feature exists to catch. Issue #824's acceptance criteria were revised accordingly (see the issue's pinned comment) before this build started.

A second, separate problem surfaced during design: several of the spike's reference review units are spawns of the real `code-reviewer` / `security-reviewer` agents against **real, already-merged historical PRs in `me2resh/apexyard`**. Those agents are hard-wired (see `.claude/agents/code-reviewer.md` § "HARD STOP") to call `gh pr view`, post a live review comment via `tracker_review_submit`, and write a real approval marker keyed on the PR number — all real side effects against production GitHub state. An eval harness that spawns these agents against real historical PR numbers would spam old merged PRs with eval-only review comments and pollute `.claude/session/reviews/` with stale markers.

## Options Considered

| Option | Pros | Cons |
|---|---|---|
| (a) Prose rubric — an LLM-judge scores review *text* against a 0-4 rubric, calibrated against recorded verdicts (#824's original AC) | Cheap; single judge call per entry; matches the issue's original spec | **Proven at-chance on the load-bearing question by spike #825.** Rubber-stamps confident-but-wrong reviews — the exact thing this feature must catch. |
| (b) Dynamic oracle — at eval time, spawn a *stronger* independent agent to re-review the diff fresh and derive ground truth on the spot, then compare | Always current; no stale corpus | No model in the current lineup is credibly "stronger than" Rex/Hakim/Tariq — all three are already pinned to `opus`, the framework's capability ceiling (AgDR-0050). Self-grading risk if oracle and agent-under-test are the same tier. Non-reproducible: the oracle's own judgment could vary run to run, undermining the regression-detection use case #824 exists for. 2x agent spawns per entry. |
| (c) Frozen ground truth — establish the defect set **once, offline**, from real re-review disagreements (an independent reviewer catching what another missed on the identical commit) and confirmed fixes (a defect that was actually fixed proves it was real); store it in the corpus; score every future run against that fixed answer key | Matches the exact mechanism spike #825 found *did* work (a second independent review of the same commit reliably caught the misses); reproducible — same ground truth every run means score deltas are real regressions, not judge noise; cheap per run — only the agent-under-test is spawned, no second live judge call; sidesteps the "no stronger model available" problem entirely because the human adjudicates once, offline, with time to think | Corpus curation is manual, human-adjudicated work; ground truth needs periodic refresh as the framework evolves; a small starter corpus is directional evidence, not a significance test (same caveat the spike itself carries) |

## Decision

Chosen: **(c) — frozen ground truth, scored by set-overlap, approve-precision as the headline metric.** This is spike #825's own recommendation #1/#2 (ground every score in independent verification of the diff; build the eval set from re-review disagreements + escaped bugs) applied directly, not re-derived from scratch.

Concretely:

1. **The corpus is the oracle, established offline.** Each corpus entry (`docs/eval-agents/corpus/<agent>.json`) freezes a PR/commit reference, a `ground_truth_defects` list (severity + location), an `oracle` block recording *how* that ground truth was established (`independent_review` — a different reviewer caught it on the identical commit; `confirmed_fix` — a later commit fixed exactly this defect, proving it was real; `no_contradiction` — no later review or fix ever contradicted a clean approval), and the `recorded_verdict` that was actually given plus whether it was justified. The starter corpus (10 entries: 6 Rex, 4 Hakim) is mined from this repo's own PR history (#792, #770, #767, #748, #686, #677, #773), reusing the spike's own review-history mining but re-shaped into the defect-set schema instead of rubric scores.
2. **The skill's job at run time is narrow: run the agent-under-test fresh on the frozen diff, then match its findings against the frozen defect set.** The matching step (caught / missed / false-alarm) is performed by the orchestrator running `/eval-agents` — this is scoring against an already-frozen answer key, not re-establishing truth, so it does not trip the self-grading concern the spike raised. That concern is specifically about *who establishes the ground truth* (never the agent being measured); it does not forbid a third party from checking a candidate's answer against an already-fixed key, which is what every eval harness's grader does.
3. **Approve-precision is the headline pass/fail metric.** `catch-rate` (overall, and a separate BLOCKING/HIGH-only cut) and `false-positive-rate` are reported alongside it. A single blended concordance score is never reported as the verdict — that number is exactly what hid spike #825's real finding at 0.83.
4. **A missed BLOCKING-severity ground-truth defect is an automatic WARN**, independent of the aggregate score — one severe miss should not average out against a pile of easy correct approvals.
5. **The optional 0-4 rubric dimensions survive, but demoted to advisory craft signal only** (spike recommendation #4) — they describe review *style* (engagement, calibration, false-positive discipline) and never gate pass/fail on their own. This keeps the AC's "per-dimension 0-4 scoring with written anchors" bullet satisfiable without resurrecting the disproven mechanism as the correctness oracle.
6. **The agent-under-test is spawned against sanitized, PR-agnostic diff text — never a live PR number.** The corpus's `pr`/`commit`/`repo` fields exist for the harness's own diff-resolution and for human traceability; they are stripped (along with `Closes #N` / `Fixes #N` trailers and any URL) before the diff is handed to the agent-under-test. The eval prompt explicitly instructs no `gh`/tracker calls and no marker writes. This is a **mitigation, not a hard sandbox** — the agent-under-test retains its normal `Bash` tool access and could in principle still attempt a tracker call if it inferred a real PR number from diff content the sanitizer missed. The skill adds a post-hoc **contamination check**: if the agent's output shows any sign it queried a live tracker (a real PR number matching the corpus entry, `gh pr` output, a live `headRefOid`), the run is discarded and flagged `CONTAMINATED` in the report rather than silently scored. Residual risk, explicitly accepted for v1; not eliminated.
7. **v1 scope is Rex/Hakim/Tariq** — all three review a diff and share the same `APPROVED / CHANGES REQUESTED / COMMENT` verdict vocabulary (confirmed against their agent files), so the defect-set-overlap scoring shape applies uniformly. **Naqid is out of scope for v1.** Naqid challenges the *premise* of an idea/plan/spec, not a diff — it has no defect-set structure to score against, and inventing one now would be exactly the "re-derive the whole method from scratch" mistake the spike warns against. The corpus schema and skill are agent-parameterized so Naqid support is a natural extension, not a rewrite, when there's a real premise-level ground-truth corpus to seed it with. The starter corpus itself only seeds Rex + Hakim (no Tariq-reviewed PRs were in the spike's sample); `tariq` is a valid `/eval-agents` argument but requires an operator-supplied `--corpus` until one is built.

## Consequences

- No CI gating in v1 — `/eval-agents` is manual/periodic measurement only, per #824's explicit out-of-scope. It is not wired to any merge-gate, pre-commit, or pre-push hook.
- Corpus quality is the whole game. The skill validates schema, not truth — a corpus entry with a fabricated or stale ground-truth defect produces a confidently wrong score. Growing the corpus is deliberately left as manual, human-adjudicated work (same cost the spike itself paid), not automated in v1.
- Approve-precision on a 6-9-entry-per-agent starter corpus is directional evidence, not a statistically significant measurement — the same limitation spike #825 states explicitly for its own n=19. Treat a WARN as "investigate," not "the model regressed," until the corpus grows.
- The Hakim starter corpus has no confirmed MISS entry (no clean Hakim-authored wrong-approval surfaced in the spike's sample) — a real, documented gap, not an oversight. Approve-precision on Hakim's starter run will trivially read 1.0 until a genuine miss is harvested and added.
- The advisory 0-4 rubric intentionally carries zero weight in the pass/fail verdict, so a future maintainer cannot accidentally slide back to "the rubric score is the answer" without a new, explicit decision.
