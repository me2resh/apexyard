# pi Gate Dispatcher — a Single-Extension Adapter Over the Bash Hooks

> In the context of making apexyard's governance gates enforceable inside pi (pi.dev) as well as Claude Code, facing the choice between N per-hook pi extensions and a re-implementation of the gate logic in TypeScript, I decided to ship one dispatcher extension that reads a gate table and shells out to the existing, unmodified bash hooks to achieve zero logic duplication across harnesses, accepting the ongoing cost of keeping the gate table in sync with `.claude/settings.json`'s hook wiring and the residual, currently-unverified risk that pi's true internal `tool_call` tool-name enumeration might drift from what's documented in its shipped `.d.ts`.

## Context

Spike me2resh/apexyard#804 proved (VIABLE verdict) that a pi extension can enforce `block-unreviewed-merge.sh` by registering on pi's `tool_call` event, reconstructing the exact stdin JSON the bash hook already parses, and mapping the hook's exit code (2 = block, 0 = allow) to pi's `{block, reason}` return contract — with zero merge-approval logic duplicated into TypeScript. The spike's own "named gaps" section flagged three follow-up decisions the real feature needed to make:

1. How to wire **more than one** gate hook without N copy-pasted single-hook extension files.
2. How to resolve an **ops root** (the directory holding `.claude/session/reviews/*.approved` markers) under pi, given that `_lib-ops-root.sh`'s session-pin mechanism (`pin-ops-root.sh` + `CLAUDE_CODE_SESSION_ID`) is Claude-Code-specific and has no pi equivalent.
3. Whether to trust the spike's hand-rolled structural types or import pi's real ones now that this is shipped code, not a throwaway prototype.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **N separate extension files, one per gate hook** | Each file is small and independently toggleable; matches the spike prototype's shape exactly | N near-identical boilerplate blocks (stdin JSON build, exec, exit-code mapping) that drift independently as pi's API evolves; adding a gate means copy-pasting a whole file instead of a table row |
| **Reimplement each gate's approval logic in TypeScript, native to pi** | No bash subprocess per tool call; could use pi-native APIs more idiomatically | Creates a second source of truth for every governance rule — exactly the problem the adapter-over-bash pattern exists to avoid. The spike ticket named this the fallback only if the pattern proved non-viable; it didn't. |
| **One dispatcher extension driven by a gate table (CHOSEN)** | One registration, one `tool_call` handler, N gates as data rows; adding a gate is a table entry, not a new file; the table is a direct visual analog of `.claude/settings.json`'s PreToolUse wiring — "one config, two harnesses" | The dispatcher itself is now the thing that must track pi's real tool-name surface accurately (see decision below); a bug in the shared dispatcher affects every gate at once, not just one |

## Decision

Chosen: **one dispatcher extension driven by a `GateDefinition[]` table** (`harness-adapters/pi/src/gate-dispatcher.ts`), because it is the shape the spike report itself recommended ("a small dispatcher extension that loops over the same hook list `.claude/settings.json` already wires for Claude Code, in the same order") and it keeps the zero-duplication property at the *gate* level (bash owns every approval decision) while collapsing the *per-gate* boilerplate to configuration.

Two supporting decisions, made while building the dispatcher:

- **Ops-root resolution**: a new, pi-only module (`harness-adapters/pi/src/resolve-ops-root.ts`) that reproduces `_lib-ops-root.sh`'s anchor-walk (the `.apexyard-fork` marker / legacy `onboarding.yaml`+`apexyard.projects.yaml` pair) faithfully in TypeScript, and substitutes an explicit `APEXYARD_OPS_ROOT` environment variable for Claude Code's session-pin mechanism (pi has no `CLAUDE_CODE_SESSION_ID` equivalent and no SessionStart hook to write a pin). This is a **deliberate divergence**: pi users get the anchor-walk safety (same as Claude Code) but not the pin-first protection that closed apexyard#381 (the `/tmp`-clone-resolves-wrong-tree bug). Consequence: a pi session launched from inside an unrelated ops-fork-shaped directory tree (e.g. a throwaway clone) could resolve markers to the wrong fork, same failure class #381 fixed for Claude Code. Mitigation available today: set `APEXYARD_OPS_ROOT` explicitly in any pi session that isn't launched from the real ops fork's working tree.
- **Real pi types, not spike shims**: `gate-dispatcher.ts` imports `ExtensionAPI`, `ToolCallEvent`, `ToolCallEventResult` directly from `@earendil-works/pi-coding-agent`, rather than the spike prototype's hand-rolled structural interface. Checking the real `.d.ts` (not just the spike's docs-based reasoning) surfaced two corrections: the result type is `ToolCallEventResult`, not `ToolCallResult` (an invented name that happened to typecheck against the shim), and pi's `ToolCallEvent` is a **discriminated union** by `toolName` (`bash` | `read` | `edit` | `write` | `grep` | `find` | `ls` | custom) — there is no `multi_edit` tool (pi's `edit` tool accepts multiple `{oldText, newText}` pairs per call), and the edit/write tools key their target as `path`, not `file_path`. That last fact turned out to need no hook change at all: `require-active-ticket.sh` and `require-migration-ticket.sh` already fall back to `.tool_input.path` when `.tool_input.file_path` is absent.

## Consequences

- Adding a new gate to both harnesses is now "add a row to `DEFAULT_GATES`", not "write a new file" — the dispatcher table and `.claude/settings.json`'s hook list should be kept in visual sync by whoever edits either.
- The `require-active-ticket` / `require-migration-ticket` mapping to pi's `edit` / `write` tool names is verified against pi's real, installed `.d.ts` (not guessed), which closes most of the residual risk the spike flagged — but the ONE thing that remains genuinely unverified is whether pi's *live internal dispatch* actually calls `tool_call` handlers with events matching this `.d.ts` during a real, model-driven agent turn (see the test suite's "LIVE" vs by-construction distinction, and AC-6 of me2resh/apexyard#815).
- The pi-flavored ops-root resolution is a new code path apexyard has to maintain in parallel with `_lib-ops-root.sh`; a future change to the bash version's anchor conditions must be mirrored here by hand (there is no shared source file across bash and TypeScript).
- Every gate hook still receives its input via a real subprocess spawn per matching tool call — the spike's own "no pre-filter, by design" note applies unchanged here; this trades a small latency cost for keeping 100% of the decision logic in one place.

## Update (GH-840)

This AgDR's own "Consequences" section named the risk plainly: *"a bug in the shared dispatcher affects every gate at once"* and *"the dispatcher table and `.claude/settings.json`'s hook list should be kept in visual sync by whoever edits either"* — the hand-maintained `DEFAULT_GATES` table this decision originally chose. AgDR-0092 (the opencode adapter) then converged on a strictly better pattern — deriving the gate table from `.claude/settings.json` at runtime, closing the drift risk by construction — and named porting pi to the same pattern as its own explicit follow-up, deliberately deferred out of that PR's scope. This section records that follow-up landing (#840 C5).

**Decision: pi now derives its gate table from `.claude/settings.json`, exactly as opencode does, sharing the harness-agnostic parsing core.** `DEFAULT_GATES` is retired.

**What changed:**

- **New shared module**: `harness-adapters/_shared/derive-gates-core.ts` — the settings.json-parsing logic (regex hook-path extraction, `Bash(<glob>)` predicate parsing, glob→RegExp, matcher-group→`GateDefinition` collapsing) factored out of the opencode adapter's `derive-gates.ts`, verified byte-for-byte identical output against this repo's own real `.claude/settings.json` before the extraction (`harness-adapters/_shared/test/derive-gates-core.test.ts` includes a cross-runtime parity test proving this). Gates here are keyed by the RAW Claude Code matcher token (`"Bash"`, `"Edit"`, `"Glob"`, …), not by either runtime's own tool vocabulary — that translation is each adapter's own job.
- **New `harness-adapters/pi/src/derive-gates.ts`** — the pi-specific translation layer over the shared core: `CLAUDE_MATCHER_TO_PI_TOOL` (with `Glob` → `find`, pi's closest analog — pi has no `glob` tool, a judgment call recorded in the module's header comment), `claudeToolNameFor`, and `findUnsupportedGateWires` (the same #840 C2 "fail loud on tools with no stdin builder" guard opencode uses, now needed here too because deriving the FULL table surfaces `suggest-mcp-search.sh`'s `Read|Glob|Grep` wiring, which this adapter has no verified pi stdin shape for — `read`/`grep`/`find`/`ls` are deliberately not guessed, matching the same discipline C2 established).
- **`harness-adapters/pi/src/gate-dispatcher.ts` rewritten**: `DEFAULT_GATES` and the old per-gate `buildToolInput` field are gone; a single generic `buildPiToolInput` (mirroring opencode's `buildToolInput`) reconstructs stdin for `bash`/`edit`/`write`. The dispatcher now calls `deriveGatesFromOpsRoot` — a pi-local sibling of opencode's identically-named function, including the same #840 C3 fix (a settings.json PARSE failure warns to stderr; a MISSING file stays silent) — **per `tool_call` event**, not once at registration. This is a deliberate, structural difference from opencode kept from the original design: pi hands the dispatcher a fresh `ctx.cwd` on every event (this AgDR's original rationale for per-call ops-root resolution); re-deriving the gate table on the same cadence means a live edit to `.claude/settings.json` mid-session is picked up on the very next tool call, at the cost of one extra JSON parse per call (negligible next to the subprocess spawn(s) that follow). The "unsupported gate wire" warning is deduped per `(opsRoot, gate, tool)` via a process-lifetime `Set` (`warnOnce`) so a Read/Glob/Grep-heavy session doesn't print the same warning before every matching call.
- **`harness-adapters/pi/test/derive-gates.test.ts`** (new) — mirrors `harness-adapters/opencode/test/derive-gates.test.ts`'s structure and coverage, adapted to pi's tool vocabulary; includes a drift test against this repo's real `.claude/settings.json` proving every hook the old `DEFAULT_GATES`/README.md "Known NOT-yet-bridged gates" table explicitly deferred (`validate-branch-name.sh`, `validate-pr-create.sh`, `require-skill-for-issue-create.sh`, `block-onboarding-in-git.sh`, and others) is now derived automatically.
- **`harness-adapters/pi/test/gate-dispatcher.test.ts` updated**: existing synthetic/LIVE cases ported to the new `wires`-shaped `GateDefinition`; new cases added for real settings.json-derived dispatch end-to-end (including a mid-"session" settings.json edit being picked up on the next call — the concrete behavior the per-call design buys), the parse-failure warning, and the deduped unsupported-tool warning. The two LIVE cases (real `gh` state against `me2resh/apexyard`) were re-run against this refactor and still pass unchanged — derivation now sources `block-unreviewed-merge.sh` from the real settings.json instead of the old curated table, with identical observable behavior.
- **`harness-adapters/pi/README.md`** — the "What this enforces" curated-table framing and the "Customizing the gate table" code example (which referenced the now-deleted `DEFAULT_GATES` export) updated to match; "Known gaps / what's unverified" gained the two new caveats (unsupported-tool warning, per-call re-derivation trade-off).

**What did NOT change:** `harness-adapters/opencode/src/derive-gates.ts` was refactored to call the new shared core too (previously self-contained), verified against its full existing 52-test suite plus the live `test/smoke-block-unreviewed-merge.sh` proof — zero behavior change, same gate table, same field names, same `findUnsupportedGateWires`/timeout/parse-warning behavior from #840 C1–C4. This was a "prove it's safe, don't just claim it" refactor: the cross-runtime parity test in `derive-gates-core.test.ts` asserts pi's and opencode's shared-core output produce the identical gate-name set against this repo's real settings.json.

**Consequence.** The drift risk this AgDR's own "Consequences" section named — *"a future change to the bash version's anchor conditions must be mirrored here by hand"* (about ops-root resolution, still true and unchanged) and the gate-table drift risk (now closed) — is narrowed to exactly the ops-root resolution logic, which remains genuinely runtime-specific (pi's per-call `ctx.cwd`-driven walk-up vs. opencode's once-at-init `PluginInput.directory`-driven walk-up) and was never a candidate for the shared core. The gate *table* itself can no longer drift from `.claude/settings.json` for either runtime — both now derive it, both share the parser that does so.

## Update (GH-844)

**The documented install layout crashed a real pi session at startup, live, confirmed against pi 0.80.3 during #840's conformance work.** The install instructions this AgDR's own artifacts (`harness-adapters/pi/README.md`) shipped — `cp gate-dispatcher.ts resolve-ops-root.ts .pi/extensions/` — put `resolve-ops-root.ts` (and, after the #840 C5 update above, `derive-gates.ts`) directly in the directory pi's local-extension discovery scans. Neither file has a default export; pi requires every file it discovers there to export a valid factory function, so the session failed outright: `Error: Failed to load extension ".../resolve-ops-root.ts": Extension does not export a valid factory function`. No gate was enforced, silently — the opposite of this adapter's purpose. The identical bug, independently confirmed, hit the opencode adapter (AgDR-0092) the same day, for a related but distinct reason (see that AgDR's own Update section).

**Root cause.** `gate-dispatcher.ts` itself was never the problem — it has always had a clean default export, and pi's loader only inspects a discovered file's `default` export (unlike opencode's loader, which inspects every named export too — see AgDR-0092's Update section for that distinction). The problem was purely that `gate-dispatcher.ts`'s SIBLING helper modules got copied into the SAME directory pi scans, and those siblings have no default export at all.

**Decision: ship the adapter as a subdirectory, not a flat file drop, with a re-export shim as the sole discovered entry.** `src/index.ts` — `export { default } from "./gate-dispatcher.ts";` — is now the file pi is meant to discover. `bin/install-pi-adapter.sh` installs `index.ts` plus `gate-dispatcher.ts`, `derive-gates.ts`, `resolve-ops-root.ts`, and a sibling copy of the shared `harness-adapters/_shared/derive-gates-core.ts` (rewriting that one file's shared-core import path from the repo-relative `../../_shared/derive-gates-core.ts` to the installed, flat `./derive-gates-core.ts`) into ONE subdirectory — `.pi/extensions/apexyard/` by default. pi's subdirectory discovery only loads `index.ts` per subdirectory (verified live against pi 0.80.3), so the helper files sit there as ordinary relative imports the discovery scan never independently visits.

The shim shape (`export { default } from ...`, exposing nothing else) is not strictly required for pi alone — pi only ever inspected `default`, so `gate-dispatcher.ts` itself, alone in a directory, would have loaded fine. It is shipped anyway so pi and opencode ship the identical fix shape: opencode's loader treats EVERY named export as a candidate plugin, so the shim is load-bearing there. One mechanism, two runtimes, per this ticket's design.

**What did NOT change:** no dispatch/gate-evaluation logic in `gate-dispatcher.ts`, `derive-gates.ts`, `resolve-ops-root.ts`, or `harness-adapters/_shared/derive-gates-core.ts` — this is purely an install-layout and documentation fix. `bin/install-pi-adapter.sh`'s one piece of generated content is the rewritten import specifier in the installed `derive-gates.ts` copy; the shared core's own file is copied byte-for-byte.
