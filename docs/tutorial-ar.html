1. إيه هو ApexYard؟

ApexYard هو فريم ورك لإدارة المشاريع البرمجية مبني على Claude Code. تخيل إن عندك فريق كامل (مدير منتجات، مهندس برمجيات، مراجع أمني، مختبر جودة...) كلهم موجودين جوا Claude Code وبيشتغلوا معاك بنظام صارم من أول الفكرة لحد ما المنتج يطلع للمستخدمين.

الفكرة الأساسية

ApexYard بيحول Claude Code من مجرد مساعد برمجي لـفريق تطوير كامل بيتبع دورة حياة برمجية منظمة (SDLC)، وبيفرض القواعد تلقائيًا من خلال hooks و gates.

ليه تستعمل ApexYard؟

  • نظام وانضباط — بدل ما تكتب كود عشوائي، كل شغل بيمر بدورة حياة كاملة
  • مراجعة تلقائية — كود ريفيور آلي (Rex) بيراجع كل PR
  • توثيق تلقائي — كل قرار تقني بيتسجل في AgDR، وكل ميزة ليها PRD
  • إدارة مشاريع متعددة — تقدر تدير أكتر من مشروع من مكان واحد
  • حماية من الأخطاء — hooks بتمنعك تعمل حاجات غلط زي push على main مباشرة

2. هيكل المشروع

لما تعمل fork من ApexYard، هيكون عندك الهيكل ده:

apexyard/                          ← الفورك بتاعك (الـ "ops repo")
├── CLAUDE.md                       ← الملف الرئيسي اللي Claude بيقرأه
├── onboarding.yaml                 ← إعدادات شركتك والتيك ستاك
├── apexyard.projects.yaml          ← قائمة المشاريع المُدارة
│
├── .claude/                        ← القواعد والأوامر والحراسة
│   ├── hooks/                      ← 18 سكريبت حراسة تلقائية
│   ├── rules/                      ← 8 ملفات قواعد
│   ├── skills/                     ← 33 أمر (slash command)
│   ├── agents/                     ← وكلاء متخصصين (code reviewer, security...)
│   └── settings.json               ← ربط الـ hooks بالأحداث
│
├── roles/                          ← 19 تعريف دور
├── workflows/                      ← سير العمل (SDLC, code review)
├── templates/                      ← قوالب (PRD, tech design, AgDR)
├── golden-paths/                   ← CI/CD pipelines جاهزة
│
├── workspace/                      ← نسخ المشاريع الحية (متتبعش بالـ git)
│   └── my-saas-app/                ← git clone للمشروع الفعلي
│
└── projects/                       ← وثائق المشاريع (متتبع بالـ git)
    └── my-saas-app/
        ├── README.md
        └── roadmap.md
الفرق بين workspace و projects

workspace/ = الكود الفعلي اللي بتشتغل عليه (gitignored — كل مشروع ليه remote خاص بيه)
projects/ = الوثائق والخطط والتقارير عن المشروع (متتبعة في الفورك بتاعك)

المرحلة 1: التخطيط (Planning)

الدور: Tech Lead

في المرحلة دي بتحدد إيه اللي هتبنيه وليه. لو المشروع محتاج PRD (وثيقة متطلبات المنتج)، بتكتبها هنا.

  • مراجعة المتطلبات وتوضيحها
  • تقدير الوقت المطلوب
  • إنشاء التذاكر (GitHub Issues)

الخروج: المتطلبات واضحة + التذاكر موجودة ومرتبة

المرحلة 2: التصميم التقني (Technical Design)

الدور: Tech Lead

بتكتب التصميم التقني — إيه المكونات، إزاي بتتكلم مع بعض، إيه الـ API، إيه الـ database schema.

  • كتابة وثيقة التصميم التقني
  • تقسيم الشغل لمهام صغيرة
  • تحديد المخاطر
  • تسجيل القرارات التقنية في AgDR

الخروج: التصميم معتمد + المهام متقسمة + AgDR لكل قرار

المرحلة 3: البناء (Build)

الدور: Backend/Frontend Engineer

قاعدة صارمة: تذكرة واحدة في المرة

اشتغل على تذكرة واحدة بس. خلصها بالكامل قبل ما تبدأ اللي بعدها. كل PR = تذكرة واحدة.

خطوات البناء:

  1. فعّل التذكرة: /start-ticket 42
  2. أنشئ branch: feature/#42-add-auth
  3. اكتب الكود + الاختبارات
  4. شغّل lint + typecheck + test + build محليًا
  5. أنشئ PR

الخروج: الكود جاهز + الاختبارات ناجحة + PR مفتوح

المرحلة 4: المراجعة (Code Review)

الدور: Tech Lead + Rex (المراجع الآلي)

Rex (المراجع الآلي) بيراجع كل commit تلقائيًا. بيشيك على:

  • هل الكود بيتبع المعمارية الصح
  • هل في اختبارات كافية
  • هل في مشاكل أمنية
  • هل في AgDR لو في قرار معماري
  • هل الـ PR فيه Glossary

الخروج: موافقة المراجع + CI أخضر

المرحلة 5: اختبار الجودة (QA)

الدور: QA Engineer

محطة إجبارية

التذكرة ما بتتقفلش بعد merge. لازم QA يتحقق من كل acceptance criteria الأول.

بعد merge، التذكرة بتتحرك لحالة QA مش Done. QA بيتحقق من:

  • كل معايير القبول متحققة
  • الحالات الطرفية (edge cases) شغالة
  • مفيش regression في الحاجات القديمة

المرحلة 6: النشر (Deploy)

الدور: Platform Engineer

  • اختبار على staging
  • خطة rollback جاهزة
  • المراقبة متفعلة
  • نشر على production

المرحلة 7: المراقبة (Monitor)

الدور: SRE

بعد النشر، بتراقب لمدة 24-48 ساعة. لو في مشكلة، بتتصرف فورًا.

4. مثال عملي كامل: من الفكرة للمنتج

خلينا نمشي مع بعض خطوة بخطوة. تخيل إنك عايز تضيف نظام تسجيل دخول لتطبيقك.

الخطوة 1: سجّل الفكرة

/idea

اكتب فكرتك. الأمر ده بيضيفها في projects/ideas-backlog.md.

الخطوة 2: اكتب المتطلبات (PRD)

/write-spec

أو اكتبها يدوي باستخدام القالب في templates/prd.md. المتطلبات بتشمل:

  • المشكلة: "المستخدمين مش قادرين يسجلوا دخول"
  • user stories: "كمستخدم، عايز أسجل دخول بالإيميل عشان أوصل لحسابي"
  • معايير القبول: "المستخدم يقدر يسجل دخول بالإيميل والباسورد"

الخطوة 3: أنشئ تذكرة (GitHub Issue)

/feature

الأمر ده بيسألك عن تفاصيل الميزة وبينشئ GitHub Issue مع user story و acceptance criteria.

أو يدوي:

gh issue create --title "[Feat] Add email authentication" \
  --body "As a user, I want to log in with email..."

الخطوة 4: سجّل القرارات التقنية

/decide

قبل ما تكتب أي كود، لازم تسجل قراراتك التقنية. مثلاً:

  • ليه اخترت Better Auth بدل NextAuth؟
  • ليه اخترت NeonDB بدل Supabase؟

الأمر ده بينشئ AgDR (Agent Decision Record) في docs/agdr/.

الخطوة 5: ابدأ الشغل

/start-ticket 5

الأمر ده بيفعّل التذكرة #5 للجلسة الحالية. بدون الخطوة دي، الـ hooks مش هتسمحلك تعدل أي ملفات كود.

بعدها أنشئ branch:

git checkout -b feature/#5-add-email-auth

الخطوة 6: اكتب الكود

اكتب الكود بتاعك واتبع المعايير:

  • TypeScript strict mode
  • اختبارات (تغطية > 80%)
  • مفيش any من غير تبرير
  • مفيش أسرار مكتوبة في الكود

الخطوة 7: شغّل الفحوصات محليًا

npm run lint && npm run typecheck && npm run test && npm run build

لازم كلهم يعدوا قبل ما تعمل push. الـ hook بتاع pre-push-gate.sh هيفكرك بده.

الخطوة 8: ارفع الكود وأنشئ PR

git push -u origin feature/#5-add-email-auth

بعدها أنشئ PR (عنوان الـ PR لازم يتبع الفورمات):

gh pr create --title "feat(#5): add email authentication" \
  --body "## Summary
- Add email/password authentication using Better Auth
- Add login and signup pages

## Testing
1. Run npm test
2. Visit /login and try signing in

Refs #5

---

## Glossary
| Term | Definition |
|------|------------|
| Better Auth | مكتبة مصادقة لـ TypeScript |
| Session token | رمز جلسة يحفظ حالة تسجيل الدخول |"

الخطوة 9: مراجعة الكود

Rex (المراجع الآلي) بيراجع تلقائيًا بعد كل commit. لو عايز تشغله يدوي:

/code-review

لو في ملاحظات، صلّحها وادفع commits جديدة. Rex بيراجع تاني تلقائيًا.

الخطوة 10: الموافقة على الدمج

بعد ما Rex يوافق و CI أخضر، لازم تأخذ موافقة صريحة على الدمج:

/approve-merge 5
موافقة الدمج لازم تكون صريحة

كلمة "go" أو "ship it" في سياق خطة عامة مش كافية. لازم تقول بوضوح "approved" أو تستعمل /approve-merge لكل PR على حدة.

الخطوة 11: اختبار QA

بعد الدمج، التذكرة بتتحرك لـ QA مش Done. اتحقق من كل acceptance criteria:

  • المستخدم يقدر يسجل حساب جديد
  • المستخدم يقدر يسجل دخول
  • المستخدم يقدر يسجل خروج
  • رسالة خطأ لو الباسورد غلط

الخطوة 12: النشر والمراقبة

انشر على Cloudflare وراقب لمدة 24 ساعة. لو كل حاجة تمام، التذكرة بتتقفل.

6. الحراسة التلقائية (Hooks)

الـ Hooks هي سكريبتات بتشتغل تلقائيًا قبل أو بعد أي عملية في Claude Code. بتمنعك تعمل حاجات غلط وبتفرض القواعد من غير ما تحتاج تفتكرها.

الحراسات الوقائية (بتمنعك من أخطاء)

الـ Hookبيعمل إيه
require-active-ticket.sh بيمنعك تعدل ملفات كود من غير ما يكون عندك تذكرة مفعلة بـ /start-ticket
block-main-push.sh بيمنع الـ push مباشرة على main — كل تغيير لازم يكون من خلال PR
block-git-add-all.sh بيمنع git add -A أو git add . — لازم تضيف ملفات محددة
block-unreviewed-merge.sh بيمنع الدمج من غير مراجعة Rex + موافقة صريحة
block-merge-on-red-ci.sh بيمنع الدمج لو CI فاشل — حتى لو الفشل مش متعلق بالـ PR
check-secrets.sh بيبحث عن أسرار مكتوبة في الكود (API keys, passwords, tokens)

حراسات التنسيق (بتضمن الشكل الصحيح)

الـ Hookبيعمل إيه
validate-branch-name.sh بيتأكد إن اسم الـ branch بيتبع الفورمات: type/#ID-description
validate-pr-create.sh بيتأكد إن عنوان الـ PR بيتبع: type(#ID): description
validate-commit-format.sh بيتأكد إن رسالة الـ commit بتبع الفورمات الصحيح
verify-commit-refs.sh بيتأكد إن أرقام التذاكر المذكورة في الـ commit موجودة فعلاً

حراسات متخصصة

الـ Hookبيعمل إيه
require-agdr-for-arch-changes.sh بيطلب AgDR لو في تغييرات معمارية
require-design-review-for-ui.sh بيطلب مراجعة تصميم لو الـ PR بيأثر على الواجهة
require-migration-ticket.sh بيطلب تذكرة migration مخصصة + AgDR قبل تعديل ملفات migration
auto-code-review.sh بيشغل Rex تلقائيًا بعد كل commit جديد

check-upstream-drift.sh بيخبرك لو الفورك بتاعك متأخر عن upstream
onboarding-check.sh بيشيك لو onboarding.yaml لسه فيه قيم افتراضية
suggest-ticket-template.sh بيقترح القالب المناسب للتذكرة

7. الأدوار (Roles) — 19 دور

ApexYard فيه 19 دور متخصص. كل دور ليه ملف تعريف بيحدد مسؤولياته، إيه يقدر يعمله، وإيه مش مسموح بيه. الأدوار بتتفعل تلقائيًا حسب السياق.

الهندسة (7 أدوار)

الدوربيتفعل لما
رئيس الهندسةمراجعة معمارية، إضافة تقنية جديدة
قائد تقني (Tech Lead)تصميم تقني، تخطيط، مراجعة كود
مهندس Backendكتابة كود الخادم، APIs، قاعدة البيانات
مهندس Frontendكتابة كود الواجهة، المكونات
مهندس جودة (QA)بعد الدمج — التحقق من معايير القبول
مهندس منصاتتغييرات CI/CD، أدوات المطورين
SREحوادث الإنتاج، مراقبة، تنبيهات

المنتج (3 أدوار)

الدوربيتفعل لما
رئيس المنتجترتيب أولويات الخريطة، قرارات استراتيجية
مدير منتجكتابة PRD، user stories، معايير قبول
محلل منتجأبحاث السوق، تحليل المنافسين

التصميم (3 أدوار)

الدوربيتفعل لما
رئيس التصميمنظام التصميم، معايير UX
مصمم UIمواصفات المكونات، design tokens
مصمم UXتدفقات المستخدم، wireframes

الأمن (3 أدوار)

الدوربيتفعل لما
رئيس الأمناستراتيجية الأمان، نمذجة التهديدات
مدقق أمنيPR بيلمس auth / crypto / secrets / بيانات مستخدمين
مختبر اختراقاختبارات نشطة، اكتشاف ثغرات

البيانات (3 أدوار)

الدوربيتفعل لما
رئيس البياناتاستراتيجية التحليلات، حوكمة البيانات
محلل بياناتاستعلامات SQL، لوحات المعلومات
مهندس بياناتETL، نمذجة البيانات، schema
التفعيل للمطور المنفرد

كمطور منفرد، أنت بتلبس كل القبعات. Claude بيتعامل معاك كالتالي:

  • في مرحلة التخطيط → بيتكلم كـ Tech Lead
  • في مرحلة البناء → بيتكلم كـ Backend/Frontend Engineer
  • لو لمست auth → بيفعل المدقق الأمني تلقائيًا
  • بعد merge → بيفكرك بدور QA

8. البوابات (Gates) — نقاط التفتيش

البوابات هي نقاط تفتيش إلزامية بين المراحل. لو بوابة فشلت، توقف وأكمل الخطوة الناقصة.

البوابةقبلتحقق من
1 PRD → تصميم تقني PRD معتمد، التذكرة الأم موجودة
2 تصميم → بناء التصميم معتمد، تذاكر القصص موجودة، AgDR للقرارات
3 بداية الكود التذكرة موجودة ومفعلة، الـ branch جاهز
3a تعديل migration تذكرة migration + AgDR migration (عبر /migration)
4 إنشاء PR الاختبارات ناجحة، تغطية > 80%، AgDR مرتبط
5 دمج PR مراجعتين (Rex + موافقة صريحة)، CI أخضر، SHA متطابق
6 التذكرة → Done QA تحقق وأعطى الموافقة
القاعدة الذهبية

لو بوابة فشلت → توقف. أكمل الخطوة الناقصة الأول. مفيش استثناءات — حتى لـ "مجرد إصلاح بسيط".

10. سجل القرارات (AgDR)

Agent Decision Record هو وثيقة بتسجل كل قرار تقني مهم. ليه اخترت المكتبة دي؟ ليه الطريقة دي وليس التانية؟ الإجابات بتتسجل في AgDR عشان مستقبلك يفهم ليه.

امتى لازم تعمل AgDR؟

لو لقيت نفسك بتعمل أي حاجة من دول → توقف واستعمل /decide:

  • "هستعمل Better Auth" → توقف، سجل ليه
  • "هختار بين X و Y" → توقف، سجل المقارنة
  • إضافة مكتبة جديدة → توقف
  • تغيير معماري → توقف
  • اختيار نمط تصميم (design pattern) → توقف

شكل AgDR

# اختيار Better Auth للمصادقة

> في سياق بناء نظام مصادقة لتطبيق SaaS،
> أمام الحاجة لمكتبة مصادقة بسيطة وقوية،
> قررت استعمال Better Auth لتحقيق مصادقة آمنة وسريعة،
> مع قبول إن المجتمع أصغر من NextAuth.

## السياق
بنبني تطبيق SaaS على Cloudflare Workers ومحتاجين مصادقة.

## الخيارات

| الخيار | المميزات | العيوب |
|--------|----------|--------|
| Better Auth | بسيط، TypeScript أصلي، يشتغل مع Cloudflare | مجتمع أصغر |
| NextAuth | مجتمع كبير، موثق كويس | مربوط بـ Next.js |
| Lucia | خفيف، مرن | صيانة أقل |

## القرار
اخترنا **Better Auth** لأنه TypeScript أصلي وبيشتغل مع Cloudflare Workers.

## النتائج
- المصادقة هتكون أسرع في الإعداد
- لازم نكتب بعض الأكواد المساعدة بنفسنا

AgDRs بتتحفظ في: docs/agdr/AgDR-NNNN-slug.md داخل مشروعك.

  1. افتح الفورك: cd ~/apexyard
  2. شوف إيه محتاج انتباهك: /inbox

  4. روح لمشروعك: cd workspace/my-app/

  6. ارجع للفورك: cd ~/apexyard

  1. تذكرة قبل الكود — /start-ticket مطلوب دائمًا

  3. PR بعنوان صحيح — feat(#42): description
  4. Glossary في كل PR — حتى لو سطر واحد
  5. AgDR لكل قرار تقني — /decide
